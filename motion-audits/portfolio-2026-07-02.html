portfolio — BookingModal motion audit

New self-hosted call-booking dialog on a terminal/CRT-styled developer portfolio. Entrance motion inherited from the site's DemoModal; the day/time pickers, form reveal, and confirmation states are new surface.

Astro + Preact island · CSS-only motion · tokens from tokens.css · Europe/Lisbon booking windows

4 findings 1 critical 3 important 3 opportunities

Overall

The window chrome is right: the enter animation, tokens, and reduced-motion handling are lifted verbatim from the site's proven DemoModal, so the modal lands on-brand. The gap is everything after it opens — every state change inside the flow (loading→slots, picking a slot, submitting, confirming, closing) is an instant cut with no transition. The bones are polished; the flow is unpolished.

Primary

Concern — the funnel's state changes lack the 150–200ms of care that separate "works" from "feels trustworthy".

What's working well

  • Entrance reuses the DemoModal recipe verbatim — same tokens, curves, and 8px rise; system consistency over novelty — BookingModal.css:31–33
  • Duration/easing come from design tokens, not ad-hoc values — var(--duration-slow) var(--ease-out-expo)
  • Backdrop fade (150ms) subtler than panel (320ms) — correct hierarchy between layers

CriticalJakubMotion gap

Booking form snaps into existence when a time is picked

WhatSelecting a slot conditionally renders the entire name/email form ({slot && <BookingForm/>}) below the time grid with no transition — a full block of new UI appears in one frame and shifts the panel height.

Why it mattersThis is the primary conversion step of the funnel. An unannounced layout jump right after the user commits to a time reads as glitchy exactly when the UI needs to feel most trustworthy.

Recommended motionFade + 6px rise on the form container: opacity 0→1, translateY(6px)→0, 180ms, var(--ease-out-quart), via a one-shot CSS animation on mount. Honor reduced motion (site pattern already does).

Day/time chips have zero hover or selection transition

WhatThe :hover and .is-active rules change color, border-color, and text-shadow instantly — no transition property exists on .bookmodal__day / .bookmodal__time (nor on the close button).

Why it mattersJakub's most-repeated polish rule: even 150ms on hover states reads as more considered than instant snaps. These chips are the most-touched elements in the modal.

Flow states hard-cut: loading → slots and form → confirmation

WhatThe loading line is replaced by the full day/time grid in one frame (if (!slots) return …), and on success the form is replaced by the confirmation block the same way (if (phase === 'done') return …).

Why it mattersLoading→content is the modal's "curtain up" — an instant swap wastes the arrival. The confirmation swap is worse: the user's payoff moment renders with less ceremony than the modal's own backdrop.

Recommended motionOne shared one-shot enter (fade + 4px rise, 150ms, quart) applied to the swapped-in container on mount — same keyframes reused for slots grid and confirmation block. Exits stay instant (subtler than enters, per Jakub).

ImportantJakub

Dialog vanishes in a single frame on close

WhatsetOpen(false) unmounts the whole dialog instantly. Enter is a considered 320ms; exit is 0ms. (DemoModal shares this behavior — it's a site-wide precedent, not unique to this component.)

Why it mattersExits should be subtler than enters — but "subtler" isn't "absent". A one-frame disappearance of a full-screen surface is the kind of cut users register as abrupt without knowing why.

Recommended motion140ms fade + 4px drift on a closing state before unmount (CSS class + animationend/timeout). If adopted, apply to DemoModal too so both windows share one exit grammar. Lowest priority of the four — precedent makes it a system decision, not a bug.
